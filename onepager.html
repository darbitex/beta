What is Darbitex Beta?

Darbitex Beta is an automated market maker on Aptos with a radical simplification of the classic AMM model. Pools are truly immutable after creation. Hooks are NFTs minted atomically at pool birth, not auctioned post-hoc. Every operation lives at the pool-primitive level for maximum composability — satellite packages can build on top without a core upgrade.

Beta is a clean-slate successor to Darbitex Alpha (V1). V1's auction-based hook system shipped with known HIGH findings in its lifecycle. Rather than patch the buggy paths in place, Beta rebuilds the core from scratch to eliminate the entire class of attacks. V1 Alpha remains frozen as legacy at a separate address.

Why 1 BPS?

Total swap fee is 0.01% — that's 30x cheaper than Uniswap V2 (0.3%). Flash loans charge the same 1 bps rate. Designed for high-frequency, high-volume trading where fee sensitivity matters.

Fee split: approximately 0.9 bps to LP providers via a per-share accumulator (MasterChef V2 pattern), and 0.1 bps to the hook pot — split 50/50 between the two hook NFTs attached to every pool. There is no separate protocol fee stream. Treasury revenue comes exclusively from holding the treasury-slot hook NFT.

Hook NFTs — No Auctions

Beta eliminates V1's auction system entirely. Instead, every pool is created with two HookNFT objects minted atomically:

Slot 0 — Treasury soulbound. Minted to the Darbitex treasury multisig at pool creation, permanently non-transferable via disable_ungated_transfer. Earns 50% of hook fees forever.

Slot 1 — Escrow tradable. Minted into the factory escrow at a fixed 100 APT listing price (admin-configurable for future pools, immutable per-listing). Anyone can call buy_hook to purchase it. After purchase, the buyer owns the NFT and earns the other 50% of hook fees. Natural price discovery emerges from accumulating unclaimed fees — the longer a pool runs without a buyer, the more the NFT is worth.

Immutable Pools

Zero admin surface at the pool level. After create_canonical_pool returns, no function anywhere in Beta can alter the pool's fee, curve, pair, or hook assignment. There is no pause, no unpause, no withdraw_protocol_fee, no admin_force_remove_hook.

The only admin surface in the entire package is set_hook_price, which updates the default listing price for future pools — existing listings keep their locked-in price. One knob, future-only effect.

LP as NFT

LP positions are Aptos objects (NFTs), not fungible coins. Each deposit mints a new LpPosition object owned by the provider. Fee math uses the MasterChef V2 pattern: a global lp_fee_per_share accumulator on the pool, and a per-position debt snapshot on the NFT.

Claim without withdraw — LPs can harvest accumulated fees at any time without touching their principal position. Remove liquidity burns the NFT and returns proportional reserves plus unclaimed fees in one shot. Positions are freely transferable (send LP to a multisig or treasury if you want).

add_liquidity uses a Uniswap V2 router-style optimal amount computation — if you provide a slippage buffer on one side, only the optimal amount is withdrawn and the unused buffer stays in your wallet.

Universal Flash Loans

Every pool supports flash loans at the same 1 bps rate. Flash receipts are hot-potato structs (no drop / store / key abilities) — must be consumed via flash_repay in the same transaction or the TX aborts. The k_after >= k_before invariant is checked in u256 to prevent rounding attacks. The pool.locked reentrancy guard blocks all swap / LP / flash operations during the borrow span.

Architecture — 3 core modules

Module Role
pool Pool struct, LpPosition (inline), HookNFT (inline), swap, LP, claim fees, flash loan, TWAP oracle
pool_factory Canonical pool creation, hook NFT escrow, buy_hook, pool registry
router Multi-hop execution with per-hop slippage protection

V1's hook_wrapper, bridge, and lp_coin modules are removed in Beta. Their functionality is either integrated into the core (bridge → pair-agnostic pool, lp_coin → inline LpPosition) or eliminated (hook_wrapper is obsolete without the auction system).

Two-Layer Composability

Every operation has two forms: a composable primitive (public fun, takes and returns FungibleAsset, no &signer) and an entry wrapper (public entry fun, handles deadline + store integration for direct user transactions). External satellite packages compose the primitive layer via cross-package use darbitex::pool — no core upgrade needed to build aggregators, arb bots, farming, or meta-routers on top of Beta.

The composable swap primitive doesn't take a signer because the FungibleAsset in the caller's hand is already proof of authorization — Move's type system guarantees the caller had to withdraw it from a primary store to have it.

Fully Decentralized

Darbitex Beta has no servers. The entire stack is decentralized:

Smart contracts — on Aptos mainnet, published via a 1-of-5 bootstrap multisig, threshold raising to 3-of-5 after the first pool is stable.

Frontend — hosted on Walrus (decentralized storage on Sui). No AWS, no Vercel, no single point of failure.

Backend — there is none. All state lives on-chain. Pool data, reserves, LP positions, hook accumulators — all queried directly from Aptos RPC.

Governance

Beta is published from a 1-of-5 bootstrap multisig (raising to 3-of-5 post-launch). Package upgrade_policy = "compatible" — struct layout changes would force a fresh package publish at a new address (same pattern V1 → Beta followed).

ADMIN, TREASURY, and REVENUE addresses are hardcoded as Move constants in the source — changing any requires a package upgrade, which itself requires the multisig threshold. Intentional minimum-trust design.

For Builders

Pool creators — call create_canonical_pool(creator, meta_a, meta_b, amount). Symmetric seeding enforced (amount_a == amount_b at initial deposit). You receive the initial LpPosition NFT.

Aggregators — enumerate via pool_factory::get_all_pools(), quote via pool::get_amount_out(), and compose via the public fun primitives: pool::swap() for single hops, router::swap_2hop_composable() / swap_3hop_composable() for multi-hop with per-hop slippage floors.

LPs — pool::add_liquidity() with desired amounts (auto-optimal, any buffer stays in your wallet), claim fees without withdrawing via pool::claim_lp_fees(), or exit via pool::remove_liquidity() with min_amount_a/b slippage floors.

Satellite packages — Beta's two-layer composability contract means future features (meta router with bridge registry, arb bots, farming, governance) all ship as independent packages depending on Beta core via cross-package import. No core upgrade needed.
